# iOS Maestro element-based `PERCY_REGIONS` support (v1.0)

## Problem Frame

Customers running Percy visual tests on iOS via Maestro on BrowserStack need to exclude **dynamic content** (ads, user-avatars, timestamp labels, animated counters) from visual diffs. Coordinate-based regions fall short because such elements move between runs — their pixel coordinates change even when the content that should be ignored is the same UI element. An element-based selector (`element: {id: "..."}` or `element: {class: "..."}`) is the right abstraction: it identifies the thing to ignore, not a fixed rectangle.

Who is affected: customers writing Maestro flows for iOS apps who have non-static UI regions they want Percy to ignore or re-evaluate with a different diff algorithm. They can already write `tapOn: {id: "submit-btn"}` in Maestro; they should be able to write `element: {id: "submit-btn"}` in `PERCY_REGIONS` and have Percy resolve the element rect at runtime.

Today's state: `PERCY_REGIONS` on iOS accepts coordinate-based regions only. Element-based regions are silently warn-and-skipped by the SDK. (Android has a related element-region resolver track using ADB view-hierarchy dump, but Android work — including any unified-key migration and cross-platform selector vocabulary — is **out of scope for this brainstorm** and will be addressed in a later, separately-scheduled cycle.) This brainstorm covers only iOS element-region resolution via WebDriverAgent on the BrowserStack iOS host.

## Requirements

- **R1.** `PERCY_REGIONS` accepts element-based regions on iOS. V1 supports two selector keys: `id` and `class`. At screenshot time, Percy's CLI relay resolves each selector to a pixel bounding box by querying WebDriverAgent on the BS iOS host, and sends the region to Percy's comparison-upload endpoint. Public-contract key-to-resolver mapping:
  - `id` → WDA `name` attribute (iOS accessibility identifier), `using: 'name'` in `POST /session/:sid/elements`. Planning's P1-2 experiment must also assert the semantic contract: on a raw Maestro-bundled WDA build, `using: 'name'` matches `accessibilityIdentifier` (not visible `label`). If the raw build diverges, R1 is revisited.
  - `class` → `XCUIElementType*`. Short-form (`Button`) accepted as a convenience; the **relay-side resolver** (not the SDK) normalizes by prefixing `XCUIElementType` if absent, validates against a known allowlist of XCUI type names before issuing the WDA call, and warn-skips if the value is not a recognized type. `using: 'class name'` in `POST /session/:sid/elements`. Normalization happens in exactly one place to avoid double-prefix bugs.
- **R2.** When a selector matches multiple elements on the current screen, Percy uses the first match (deterministic, matches `percy-appium-python` and the Android resolver pattern).
- **R3.** When a selector matches zero elements, Percy logs a warning naming the selector **key** and the **length** of the value (but not the value itself — selectors are customer-controlled and can carry PII like emails, UUIDs, or internal IDs; Maestro stdout is cross-tenant-observable via BrowserStack log aggregation). The region is skipped; other regions in the same `PERCY_REGIONS` array are still sent; the screenshot upload still succeeds.
- **R4.** Resolved coordinates are correctly scaled for each iPhone (points → pixels) via **width-ratio**: `raw = screenshot_pixel_width ÷ wda_window_logical_width`. Order of operations: (1) compute `raw`; (2) validate `raw ∈ [1.9, 3.1]` — if outside, fail-closed (warn-skip all element regions for the session) with a log line distinguishing "scale out of range" from other warn-skip causes; (3) snap `raw` to the nearest integer in `{2, 3}` and use that as the scale. Scale factor is cached per-Maestro-session (keyed by `sessionId`). The relay parses the PNG IHDR chunk (bytes 16–23 big-endian) to obtain `screenshot_pixel_width` — a one-time 24-byte prefix read, no new dependency. PNG dimension parsing happens **before** any WDA element-resolution calls so that landscape detection and scale validation can short-circuit WDA work. Mirrors `percy-appium-python`'s same-unit-family ratio pattern. Customers don't maintain a device catalog; new iPhones work automatically.
- **R5.** The SDK script `percy/scripts/percy-screenshot.js` forwards element-based regions to the `/percy/maestro-screenshot` relay for iOS. The current SDK pre-relay warn-and-skip gate is removed. Shape validation happens in the CLI relay (one source of truth).
- **R6.** Cross-tenant safety on BrowserStack shared iOS hosts: the CLI relay resolves each screenshot's WDA port via a **session-scoped authoritative source** written by realmobile at session start. realmobile writes `/tmp/<sid>/wda-meta.json` containing `{sessionId, wdaPort, processOwner, flowStartTimestamp}`. The relay validates the file's process owner (must be the expected Maestro process) and freshness (timestamp must post-date Percy CLI startup for the session). If the metadata file is missing, stale, or ownership-mismatched, the relay fails closed — element regions are warn-skipped for that screenshot, the upload still succeeds. The relay must only query WDA on loopback (`127.0.0.1`); non-loopback WDA addresses are refused.
- **R7.** `PERCY_REGIONS` is not a security boundary. The README explicitly states customers must not rely on element or coordinate regions to prevent PII, credentials, or regulated data from being captured in screenshots — failed selectors (R3), device rotation (see Scope Boundaries), R6 fail-closed, and Percy↔BS baseline-linkage bugs can all cause masked content to appear in uploads or baselines. Relay-enforced guardrails at resolution time:
  - Maximum selector string length: iOS path caps at 256 chars (Android-path existing 512-char cap is deliberately not changed in V1 — iOS-only tightening).
  - Per-selector WDA timeout: ~500ms (warn-skip on exceed).
  - Maximum regions-per-screenshot: reuse the existing validator cap (~50).
  - No WDA response bodies emitted to Maestro stdout on failure. Additionally, Maestro stdout logs must not carry WDA sessionId, WDA port, resolved pixel coordinates, or any WDA response body on success paths either — only selector key, value length, result status, and duration.
  - Relay validates each resolved bounding box is fully within screenshot pixel dimensions (`0 ≤ left < right ≤ width`, `0 ≤ top < bottom ≤ height`) AND has non-trivial area (`right − left ≥ 4` and `bottom − top ≥ 4` px, to reject collapsed/hidden-element silent-success rects). Failed validation warn-skips the region with a distinct log reason.
  - **Source-dump conditional hardening:** if the P1-2 experiment selects the source-dump resolution path, the following are V1 requirements (not deferred): `/session/:sid/source` response size capped at 20 MB (iOS apps with WebViews can emit large trees; 5 MB is too small); XML external-entity resolution disabled; entity-expansion limit set; parse-time budget enforced via interrupt; response-size check happens before parse begins.
- **R8.** Documentation in the Percy Maestro README covers: the V1 selector table (`id`, `class`) with a key-to-resolver mapping column, one copy-pasteable iOS example, a note that `text` and `xpath` selectors are follow-up work in V1.1, the Scope Boundaries non-goals, and the R7 security-posture disclaimer.

## Success Criteria

- **Payload-level verifiable without Percy backend fix:** an iOS Maestro customer can add `PERCY_REGIONS: '[{"element":{"id":"my-button"}, "algorithm":"ignore"}]'` to their flow, and an engineer can inspect the outbound payload Percy CLI sends to the Percy comparison-upload endpoint and confirm the region appears in the correct payload shape with the element's resolved pixel bounding box. Primary implementation "done" signal.
- **Visual-overlay spot-check (bridges payload → dashboard gap):** Percy CLI can emit, in a debug-log channel, a locally-composited PNG showing each resolved region as a red rectangle overlay on the captured screenshot. Engineers and early customers visually confirm bbox-matches-element without depending on Percy's dashboard baselining fix.
- **End-to-end dashboard-visible outcome (v1.0 GA gating):** once Percy's BrowserStack baseline-linkage limitation is resolved, the same customer sees the region as a visual overlay on the snapshot detail page and as a populated `applied-regions` field on the comparison API object. **v1.0 GA does not ship until this is verified end-to-end.**
- **Cross-tenant isolation:** on BrowserStack shared iOS hosts, Percy CLI never resolves element selectors against a WDA session owned by another tenant, even under concurrent multi-device load. Stale-session, co-tenant-port-collision, and ambiguous discovery cases all warn-skip rather than produce incorrect regions.
- **Landscape detection & warn-skip:** the relay detects non-portrait orientation (via WDA query or screenshot aspect ratio) and explicitly warn-skips element regions with a log line, rather than producing silently-wrong coordinates. A Maestro flow that rotates mid-test produces a loud log, not a subtly-off masked region.
- Adding a new iPhone model requires zero percy-maestro or CLI code changes — the width-ratio scale factor Just Works.

## Scope Boundaries

- **Out:** Additional selector keys beyond `id` and `class`. `text` (requires WDA predicate-string construction with single-quote injection-escape; security review needed) and `xpath` (unbounded customer input; DoS risk against WDA on deep UI trees, requires complexity heuristic) are named V1.1 follow-ups. V1 ships the minimum safe surface.
- **Out:** Android unified-key migration (`resource-id` → `id`) and cross-platform selector vocabulary. Deferred to a later brainstorm cycle. This brainstorm is iOS-only.
- **Out:** Landscape / device-rotation transforms. V1 supports portrait only. Landscape is **explicitly detected and rejected** (via WDA query or screenshot aspect ratio) with a warn-skip log line. No silent wrong-coordinate production. Landscape follow-up is named.
- **Out:** Percy dashboard baseline-linkage for BrowserStack-orchestrated Maestro builds. Percy treats each BS build as `branchline_first_build_empty` because BS assigns unique `parallel-nonce` per build, preventing auto-baselining. Affects all Percy+BS integrations (not percy-maestro specifically). Tracked with BrowserStack and Percy backend teams. **v1.0 GA gates on this fix** (see Success Criteria).
- **Out:** Multi-element selector composition (AND / OR of keys in one `element: {...}`). V1 requires exactly one selector key per region.
- **Out:** iOS element regions on platforms other than BrowserStack App Automate. Local `npx percy app:exec -- maestro test` execution without BrowserStack warn-and-skips element regions (no WDA to query); coord regions still work locally. README documents the local-dev-loop gap. A local linter/validator for selector yaml (`percy maestro lint-regions`) is a named future-cycle follow-up.
- **Out:** Maestro Studio / `maestro hierarchy` CLI-based resolution path. Brainstorm-verified that `maestro hierarchy` can't run alongside an active `maestro test` flow (device session held exclusively). WDA-direct (per-element or source-dump, decided by P1-2 experiment) is the only concurrent-safe runtime resolver.

## Key Decisions

- **Resolution path (pending P1-2 live experiment):** Percy CLI relay queries WebDriverAgent on the BS iOS host. Two candidate shapes — per-element (`POST /session/:sid/elements` + `GET /element/:eid/rect`) or single-source-dump (`GET /session/:sid/source`) parsed locally, matching the `adb-hierarchy.js` pattern. Decision gated on the P1-2 live experiment: if per-element calls are concurrent-safe against a Maestro-owned session (no stale-element-reference errors, no flow-completion regressions), use per-element. Otherwise, default to source-dump. **Rationale:** mirrors the Android ADB-resolver architecture already in `adb-hierarchy.js`; Maestro's own hierarchy APIs are session-exclusive.
- **V1 selector scope: `id` + `class` only.** Defer `text` and `xpath` to V1.1. **Rationale:** `id` + `class` cover the majority of element-regions use cases (accessibility identifiers, element-type masks) and carry no injection/DoS surface. `text` requires predicate-string value escaping (reviewable later). `xpath` requires a complexity heuristic and likely an opt-in env flag with a DoS disclaimer. Ship the safe minimum sooner; pay one extra minor release for the rest. Reduces V1 review load and security surface.
- **Cross-tenant safety via realmobile-written metadata (hard dependency):** realmobile writes `/tmp/<sid>/wda-meta.json` at session start containing sessionId, WDA port, process owner, and flow-start timestamp. The relay uses this as the only session-scoped authoritative source. **Rationale:** process-scan is a shared-host scan (rejected for cross-tenant leak risk); WDA `GET /sessions` is not a stock WDA endpoint on Maestro's direct-WDA build. Realmobile cooperation is the only secure mechanism. Treating it as a hard dependency means the coordination conversation starts now, not at plan time.
- **Scale factor via width-ratio, snapped to {2, 3}:** `scale = round(screenshot_pixel_width ÷ wda_window_logical_width)`, fail-closed outside `[1.9, 3.1]`. **Rationale:** same-unit-family ratio avoids the points-vs-pixels and status-bar-inclusion ambiguity. Mirrors `percy-appium-python`. No WDA-endpoint research required. Snap to integers prevents floating-point drift from producing off-by-1 rects.
- **v-number: iOS element regions ship as v1.0 marquee.** Any future Android selector-vocabulary work is tracked separately (post-v1.0, likely v1.1 or later once a dedicated Android brainstorm completes) — iOS v1.0 does not imply Android is being downgraded or held. **Rationale:** user decision. iOS carries 4 sources of uncertainty that would otherwise block Android; decoupling honors the 'independently reviewable and releaseable' principle the v0.4.0 decision already used.
- **v1.0 GA gates on Percy↔BS baseline-linkage fix.** SDK-layer work can complete on a faster cadence, but customer-visible GA waits until dashboard masking actually works. **Rationale:** shipping a feature that looks broken to customers generates support load that outweighs the time-to-market win. Visual-overlay spot-check success criterion provides internal verification during the wait.

## Dependencies / Assumptions

- **realmobile writes `/tmp/<sid>/wda-meta.json` at Maestro session start** (hard dependency, see Key Decisions). Without this, the feature cannot ship securely. Coordination with the realmobile team is a v1.0 precondition, not a planning-time discovery.
- **BrowserStack iOS host WDA accessibility on loopback (`127.0.0.1`).** Percy CLI running on the BS iOS host reaches WebDriverAgent on the per-device port (observed in 8400–8410 range in prior session Appium process args). The relay refuses non-loopback WDA addresses.
- **BrowserStack's appautomate bridge forwards `appPercy` for iOS Maestro builds (confirmed 2026-04-21).** Prerequisite that makes Percy CLI start on the host. v0.4.0 relies on this too.
- **Percy dashboard visualization of regions on iOS app comparisons is gated on the Percy↔BS baselining fix.** This gates v1.0 GA; does not block SDK-layer implementation or payload-level success criterion.
- **WDA `GET /session/:sid/window/size` exists on Maestro's WDA build (W3C-standard).** Needed for the width-ratio scale factor. Not Appium-specific. Planning should confirm as part of the P1-2 experiment setup, but failure is unlikely.

### Deferred to Planning

- **[Affects R1, Resolution path][Live experiment] P1-2: WDA concurrent-connection safety experiment.** Resolution path (per-element vs source-dump) is gated on this. Experiment scope: on a BS iOS host, trigger `POST /session/:sid/elements` and `GET /element/:eid/rect` mid-flow against a known Maestro `maestro test` run; repeat with `GET /session/:sid/source`. Compare: (a) flow-completion rate, (b) Maestro error logs / stale-element-reference error counts, (c) **selector-semantic correctness** (with a test app where `accessibilityIdentifier` differs from visible `label`, assert `using: 'name'` matches the identifier, not the label), (d) **rect-to-screenshot temporal alignment** (overlay returned rect on captured screenshot for scrolling flows; confirm per-element rect doesn't drift from screenshot time). Run under a Maestro flow doing ≥10 `tapOn` per screenshot, for ≥20 consecutive screenshots. Also piggyback a one-time probe: log `GET /session/:sid/window/size` output from ≥3 different iPhone models, confirm values match Apple's published logical-point dimensions (not device-pixel dimensions) — catches the "WDA returns pixels not points" silent-failure mode. Outcome picks the Path implementation units. Landscape detection method falls out of the same decision: per-element path uses WDA-query orientation; source-dump path uses screenshot-aspect-ratio heuristic.
- **[Affects R6][Coordination — security-gated] realmobile artifact contract.** This is not a planning-time discovery; plan owners coordinate with realmobile before implementation to nail down the full security contract: (a) exact JSON schema for `wda-meta.json`; (b) write point in realmobile's iOS Maestro session lifecycle (session start, WDA-restart rewrites, Percy CLI restart rewrites); (c) atomic write semantics (temp-file + rename within same filesystem); (d) file-permission mode (proposed: file 0600, parent dir 0700 with `O_NOFOLLOW`); (e) symlink-safety and pre-creation-race behavior (refuse-if-exists or secure-atomic-replace); (f) cleanup on session end; (g) attested source of "expected process owner" for the relay's validation check (proposed: Percy CLI's own `getuid()` at startup, verified via `fstat` on an open fd — NOT env-provided values which are tenant-influenceable); (h) relay-side runtime enforcement of loopback-only (parse address, reject non-`127.0.0.0/8`, use explicit IP URL to prevent DNS-rebinding). **Security acceptance tests for the plan:** two-tenant concurrent-write test, permission/ownership verification, atomicity-under-crash test, symlink-attack test, pre-creation-race test. No implementation starts until these are green on a staging BS iOS host.

## Alternatives Considered

- **Unified single-shape `element: {selector_type: "id", value: "foo"}` object** instead of `element: {id: "foo"}` — more explicit but verbose; reviewer brainstorm preferred the terser key-as-selector form to match Maestro's own `tapOn: {id: "foo"}` idiom.
- **Full selector composition (AND / OR)** — rejected for V1 scope; users needing composite selectors wait for V1.1 `xpath` or compose at Maestro flow level.
- **SDK writes `/percy/maestro-init` with sessionId + WDA port** — considered as a no-realmobile-dependency path. Rejected: tenant-influenceable input (env / config) cannot anchor the cross-tenant safety property R6 requires. (Full rationale in Key Decisions — "Cross-tenant safety via realmobile-written metadata.")
